## 二、第4章「数据获取与实现」— 需要新增/更新的内容

### 4.X 爬虫单IP安全策略（新增段落）

爬虫在单公网IP限制下完成 38 区县数据采集，核心技术手段：

**（1）全局速率控制**：模块级 `threading.Lock` 确保任意两次请求间隔 ≥ 2 秒，IP 级 QPS ≤ 0.5。

**（2）安全验证自动检测**：响应中检测 `<title>安全验证</title>` → 自动保存断点 → 停止当前区县 → 提示换 Cookie → 续爬不丢数据。

**（3）Cookie 翻页上限**：单 Cookie 累计 200 页后自动暂停，避免被标记。

**（4）紧急熔断**：连续 5 次 403 → 全部线程冷却 30 分钟。

**（5）交互式爬取**：启动后列出 38 区县状态（已完成/未爬），手动选择目标，逐区县顺序爬取。

**（6）列表页提取封面图**：从 `.property-content` 卡片的 `<img>` 标签提取缩略图 URL，去除裁剪参数得原图，不进详情页。

## 三、第5章「数据清洗与存储实现」— 需要新增/更新的内容

### 5.X 定时增量更新调度器（新增段落，替换原来的 crontab 建议）

系统集成 **APScheduler**，在 FastAPI 启动时自动加载，无需外部系统。

**调度配置**：
- cron 表达式：`0 2,14 * * *`（每天凌晨 2:00 + 下午 14:00）
- 每次抓取每区县前 3 页最新房源
- 防重复：上次未完成则跳过
- 超时：单次 15 分钟
- 容错：错过 15 分钟内补偿执行

**控制方式**：

| 方式 | 说明 |
|------|------|
| 环境变量 | `SCHEDULER_ENABLED` / `SCHEDULER_CRON` / `SCHEDULER_MAX_PAGES` |
| API 查询 | `GET /api/update/scheduler` |
| API 手动触发 | `POST /api/update/scheduler/trigger` |

#### 页面架构

```
/                          → Landing 落地页（数据规模 + 进入仪表盘）
/analysis                  → 数据仪表盘（核心，客户端渲染）
  ├── KPI 数据带           ← /api/stats/overview
  ├── 房价预测模型          ← /api/analysis/prediction
  ├── KMeans 聚类画像       ← /api/analysis/clustering

  └── 区县房源排名          ← /api/stats/overview

```

#### 数据流

```
浏览器 (Netlify 静态 HTML + JS)
  └─ useEffect → fetch() → FastAPI :8000 → MySQL (54,993条)
```

**增量更新体现**：爬虫更新数据 → 用户刷新页面 → fetch 到最新数据，前端无需重新构建部署。
