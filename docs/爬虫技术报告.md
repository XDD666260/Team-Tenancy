# 安居客链家爬虫技术报告

> 学年设计 · 重庆二手房分析系统 · 爬虫模块
> 负责人：胡霖 | 2026年6月

---

## 一、项目背景与目标

爬取重庆全市二手房数据，目标 5 万条以上，覆盖全部 38 个区县，用于后续数据分析与可视化（价格预测、聚类分析、地图展示等）。

数据源选型：**安居客**（主力，移动站 m.anjuke.com） + **链家**（辅助，cq.lianjia.com），两个数据源互补去重。

### 各文件详细说明

#### anjuke_fast.py（安居客快速爬虫）
- **定位**: 大规模快速采集，目标 5 万条以上
- **URL**: `https://m.anjuke.com/cq/sale/{code}/p{page}/`（移动站）
- **并行**: 6 个区县同时爬，ThreadPoolExecutor
- **代理**: 快代理隧道，`Connection: close` 强制每次请求换 IP
- **翻页**: 每页 60-71 条，页间延迟 10-20 秒
- **产出字段**: id, title, total_price, unit_price(计算), community, district, address, layout, rooms, halls, bathrooms(计算), area, orientation, tags → **缺 lng/lat/decoration/floor_desc/total_floors/build_year**
- **断点**: `data/checkpoints/anjuke_fast_{code}.json`

#### anjuke_detail.py（安居客详情爬虫）
- **定位**: 精确采集，所有字段完整
- **流程**: 列表页(移动站) → 每个房源进 2 个详情页:
  - 桌面站 `chongqing.anjuke.com/prop/view/S{id}`：拿经纬度（`<meta name="location" coord="...">`）
  - 移动站 `m.anjuke.com/cq/sale/S{id}/`：拿装修/年代/卫生间等（`<meta name="keywords">`）
- **详情页并行**: 每个区县内部用 ThreadPoolExecutor(4) 并行抓详情
- **适用**: 对已有基础数据的区县做精确补全，或小规模爬取
- **用法**: `python crawler/anjuke_detail.py [区县名]`

#### anjuke_backfill.py（字段回填）
- **定位**: 对 `anjuke_fast.py` 产出的 CSV 补全坐标、装修等缺失字段
- **匹配策略**: 标题一致 + 区县一致 + 小区名一致
- **流程**:
  1. 读取 CSV → 分析缺失字段（lng/decoration/build_year/floor_desc/bathrooms 各缺多少）
  2. 扫描移动站列表页 1-12 页 → 建立 `{title: listing_id}` 映射
  3. 按标题匹配 CSV 中的房源 → 获取对应的 listing_id
  4. 多线程（6线程）进详情页 → 获取 lng/lat/decoration/floor_desc/build_year
  5. 边补边写入 CSV（每批次保存，支持中断续补）
- **进度管理**: `data/checkpoints/backfill_progress.json`
- **用法**: `python crawler/anjuke_backfill.py [区县名|CSV路径]`

#### lianjia_spider.py（链家爬虫）
- **URL**: `https://cq.lianjia.com/ershoufang/{code}/pg{page}/`
- **方式**: 从列表页 HTML 直接解析 `.sellListContent li` 卡片
- **Cookie**: 需从浏览器获取登录态（`lianjia_token`, `lianjia_ssid`, `hip`）
- **限制**: 翻页到第 2 页即触发登录验证，每区县仅获取 30 条（1页）
- **产出**: 300 条（10 区县 × 30）

## 四、遇到的问题与解决方案

### 4.1 安居客桌面站 SPA 化

**时间**: 2026年6月13日
**现象**: 之前 working 的 requests 爬虫突然全部返回 731B 空壳 HTML（`<div id="@@xxzlGatewayUrl">`）
**原因**: 安居客将桌面站全面切换为 Nuxt.js SPA（单页应用），房源数据通过 JS 动态加载，requests 无法获取
**解决**: 全面切换到移动站 `m.anjuke.com`，移动站仍为服务端渲染，HTML 中直接包含房源数据。CSS 选择器从 `.property-content` 改为 `li.item-wrap`

### 4.2 网易易盾验证码

**时间**: 2026年6月13-14日
**现象**: 多个 IP（校园网、手机热点）访问安居客均返回"请输入验证码"页面
**原因**: 安居客使用 58同城/网易易盾 (`c.dun.163.com`) 企业级验证码，纯 IP 黑名单机制
**解决**: 
- 使用快代理隧道代理，每次请求自动换 IP
- 约 70% 的代理 IP 能通过验证（移动站网关较宽松）
- 部分远郊区县（云阳、忠县等）即使代理也全部被拦，暂时搁置

### 4.3 代理连接中断（IncompleteRead）

**时间**: 2026年6月14日
**现象**: 代理请求远郊区县（云阳、忠县等）频繁抛出 `IncompleteRead: X bytes read, Y more expected`
**原因**: 远郊区县列表页较大（900KB+），代理连接不稳定导致传输中断
**解决**:
- 增加重试次数至 10-15 次
- 增加超时时间至 60-90 秒
- 实现 `smart_get`：先直连（不用代理），失败后走代理兜底
- 部分区县（8个极偏远）代理耗尽前未能攻克

### 4.4 链家 Brotli 编码问题

**时间**: 2026年6月13日
**现象**: 爬虫获取的链家页面为乱码，BeautifulSoup 无法解析
**原因**: 爬虫声明 `Accept-Encoding: br`（Brotli），链家返回了 Brotli 压缩数据，但 Python 环境中未安装 `brotli` 库
**解决**: `pip install brotli` 后正常解压

### 4.5 链家翻页即封

**时间**: 2026年6月13日
**现象**: 链家第 1 页 30 条正常，第 2 页无论等多久（试过 124 秒）都返回 6293B 空白登录页
**原因**: 链家反爬系统检测到翻页行为模式（同一 Session 翻页 = 爬虫特征）
**解决**: 采用"多区县浅翻页"策略——每区只爬第 1 页，靠区县数量弥补总量

### 4.6 字段不完整

**时间**: 2026年6月14日
**现象**: 快速爬虫产出的 82K 数据缺 lng/lat/decoration/build_year/floor_desc
**原因**: 移动站列表页天然不提供这些字段，需进详情页获取，但详情页请求量是列表页的 60 倍
**解决**: 设计 `anjuke_backfill.py`，按标题匹配已有 CSV → 获取 listing_id → 进详情页补字段

### 4.7 断点污染

**时间**: 2026年6月12日
**现象**: 爬虫中断重启后，已完成区县被标记为"0页"，之前的数据全部丢失
**原因**: `pages_done = []` 每次从空列表开始，不继承旧 checkpoint
**解决**: 改为 `pages_done = list(checkpoint.get('pages_done', []))`，从旧记录继承

### 4.8 Windows GBK 编码崩溃

**时间**: 2026年6月12日
**现象**: emoji 打印导致 `UnicodeEncodeError: 'gbk' codec can't encode character`
**解决**: 运行命令加 `PYTHONIOENCODING=utf-8`，代码中替换 emoji 为 ASCII 文本

## 六、反思与改进方向

### 6.1 爬虫架构反思

1. **数据源选型偏差**: 初期基于桌面站编写爬虫，未预料到 SPA 化。应提前调研网站技术栈变化趋势。

2. **代理依赖过重**: 爬虫严重依赖单一隧道代理，代理耗尽后无备用方案。应准备多套代理源（或自建代理池）。

3. **字段策略失衡**: 快速爬虫和详情爬虫的关系设计偏晚，导致大量时间浪费在重爬已有数据上。应该一开始就明确"先数量后精度"的采集策略。

4. **远郊区县重视不足**: 38 个区县中，8 个极偏远区占据大量调试时间但收获为零。应先聚焦主城 20 区，偏远区县单独处理。

### 6.2 技术选择反思

1. **requests vs Playwright**: 安居客 SPA 化后一度尝试 Playwright，但 IP 封禁问题不因浏览器引擎改变而变化。纯 IP 层面的风控，换浏览器引擎无效。

2. **代理类型**: 快代理隧道是数据中心 IP，被安居客部分识别。住宅代理效果会更好，但成本更高。

3. **断点设计**: checkpoint 机制救了多次中断，但最初设计有覆盖 bug（空列表覆盖旧记录），导致一轮数据丢失。

### 6.3 改进方向

1. **代理池多样化**: 隧道代理 + 直连 + 手机热点，三种方式互补
2. **增量更新**: 实现 `update.py`，定期爬取新增房源而非全量重爬
3. **地址地理编码**: 对无坐标的房源，通过 `重庆+区县+小区名` 地理编码补充经纬度（需解决 API 被墙问题）
4. **数据库直写**: CSV 中间存储改为直接写入 MySQL，减少 I/O 开销
5. **日志系统**: 增加结构化日志，便于追踪每个区县的采集成功率和失败原因
