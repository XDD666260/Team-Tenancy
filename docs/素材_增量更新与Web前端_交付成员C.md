## 一、第4章「数据获取与实现」需要更新的内容

### 4.X 爬虫策略升级：单IP安全爬取

> **对应文档位置**：第4章中描述爬虫技术方案的部分。在原有"多线程 + 断点续爬"描述之后，新增以下内容。

**背景**：单公网IP爬取安居客时，频繁请求会触发安全验证（滑块/点击验证）。针对这一问题，爬虫进行了以下升级：

**（1）全局速率控制**

所有爬虫线程共享一个模块级请求锁，确保任意两次 HTTP 请求至少间隔 2 秒，将 IP 级请求频率控制在 0.5 QPS 以下，远低于安居客风控阈值。

**（2）安全验证自动检测与恢复**

爬虫在每次 HTTP 响应中检测 `<title>安全验证</title>` 特征。一旦发现 ——
- 自动保存当前区县的爬取进度（断点），不会丢失已爬数据
- 立即停止当前区县，不再浪费请求
- 提示更换浏览器 Cookie 后重新运行，从断点续爬

**（3）Cookie 翻页上限**

单 Cookie 累计翻页超过 200 页后自动暂停，避免 Cookie 被标记。换 Cookie 后重新运行即可从断点继续。

**（4）紧急熔断机制**

连续 5 次收到 403 禁止访问响应 → 所有线程自动冷却 30 分钟后再恢复，避免 IP 被长期封禁。

**（5）交互式爬取模式**

改为交互式：启动后列出所有区县及其爬取状态（已完成/未爬/断点），由用户手动选择爬取目标。避免之前"全自动遍历所有区县"导致单次运行请求量过大的问题。

```
───────────────────────────────────────────────────────
#    区县        分级    上限    状态
───────────────────────────────────────────────────────
1    两江新区     主城    50页    ✓ 已完成(540条)
2    渝中区       主城    50页    ✓ 已完成(1080条)
...
38   丰都县       远郊    20页    ○ 未爬
───────────────────────────────────────────────────────
已完成 3/38，待爬 35

输入区县编号或名称: 沙坪坝区
```

**（6）列表页提取房源图片**

不进入详情页（避免请求量翻倍），直接从列表页的房源卡片 `<img>` 标签提取缩略图 URL，去掉安居客图片裁剪参数后即为原图地址。

## 二、第5章「数据清洗与存储实现」需要更新的内容

### 5.X 数据增量更新调度器

> **对应文档位置**：第5章中描述"数据维护"或"增量更新"的部分。**替换**原有的"建议用户手动配置 crontab"相关描述。

系统集成了 **APScheduler 定时调度器**，在 FastAPI 后端启动时自动加载，无需外部系统（crontab / Windows 计划任务）干预。

**调度策略**：
- 默认每天 **凌晨 2:00** 和 **下午 14:00** 各执行一次增量更新
- 每次抓取每个区县的前 3 页最新房源（`--max-pages 3`）
- 通过指纹对比：新房源 INSERT、价格变动 UPDATE、30 天未出现标记为 `probably_sold`
- 所有操作记录写入 `crawl_log` 表

**防重复与容错**：
- 上一次任务未完成时自动跳过本次触发
- 错过 15 分钟内的触发仍会补偿执行
- 多次错过只执行一次（合并执行）
- 单次更新超时时间 15 分钟

**环境变量控制**：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `SCHEDULER_ENABLED` | true | 开关 |
| `SCHEDULER_CRON` | `0 2,14 * * *` | 执行时间 |
| `SCHEDULER_MAX_PAGES` | 3 | 每区县页数 |

**调度器 API**：

| 接口 | 说明 |
|------|------|
| `GET /api/update/scheduler` | 查看调度状态（cron、下次执行时间、最近日志） |
| `POST /api/update/scheduler/trigger` | 手动触发一次调度任务 |

**监控输出示例**：
```
[Scheduler] 定时调度已启动 | cron: 0 2,14 * * * | max_pages: 3
[Scheduler] 下次执行: 2026-07-05 02:00:00
```

### 6.X Web 前端数据仪表盘（新增小节）

#### （1）技术选型

| 层级 | 技术 | 版本 |
|------|------|------|
| 框架 | Next.js 15 (App Router) | 15.x |
| UI | React + TypeScript | 19.x + 5.7 |
| 图表 | Recharts | 3.9 |
| 动画 | GSAP + ScrollTrigger | 3.12 |
| 样式 | Tailwind CSS（暗色主题） | 3.4 |
| 部署 | Netlify（静态导出） | — |

#### （2）页面架构

```
Web 前端 (Next.js)
├── /                         Landing 落地页
├── /analysis                 数据仪表盘（核心页面）
│   ├── KPI 数据带
│   ├── 房价预测模型（4模型卡片 + 特征重要性柱状图）
│   ├── KMeans 聚类画像（5类卡片 + 雷达图 + 金字塔）
│   ├── 关联规则挖掘（热力散点图 + TOP10 规则表格）
│   └── 区县房源排名（可点击钻取详情）
└── /analysis/district/[name]  区县详情页（16个）
```

#### （3）数据流设计

```
浏览器 (Netlify 静态 HTML + JS)
    │  useEffect → fetch()
    ▼
FastAPI 后端 (:8000)
    │  SQL
    ▼
MySQL (54,993条)
```

前端采用**客户端渲染**模式：页面加载后，通过 JavaScript 的 `fetch()` 并行请求 4 个后端 API，获取实时数据渲染图表。后端不可用时自动降级到内置静态演示数据，页面不会白屏。

**增量更新体现**：数据更新后，用户刷新浏览器即可看到最新数据，无需重新构建或部署前端。

#### （4）调用的后端接口

| 接口 | 返回数据 | 前端用途 |
|------|---------|---------|
| `/health` | 数据总量 | 探活 + 连接状态指示灯 |

| `/api/analysis/prediction` | 4 模型指标 + 12 维特征重要性 | 预测模型卡片 + 特征对比柱状图 |
| `/api/analysis/clustering` | 5 类画像（数量/均价/面积/户型/房龄） | 聚类卡片 + 雷达图 + 金字塔图 |
| `/api/analysis/association-rules` | 50 条规则（前件/后件/支持度/置信度/提升度） | 热力散点图 + TOP10 规则表格 |

### 6.X Android App 小节 — 补充 API 变更

> **对应文档位置**：第6章 Android App 相关小节。在已有的接口列表后补充。

**新增/变更的 API 字段**：

（1）房源列表 `GET /api/houses` 每条记录新增：
```json
{
  "image_urls": ["https://pic1.ajkimg.com/...", "https://image1.ljcdn.com/..."],
  "image_url": "https://pic1.ajkimg.com/..."  
}
```
- `image_urls`：从详情页爬取的真实多图（目前链家 509 条有图）
- `image_url`：从列表页提取的封面缩略图（所有安居客房源都有）
- Kotlin 端可直接用 Glide 加载 `image_url` 作为列表封面

（2）新增筛选参数：
- `has_images=true` — 仅返回有真实图片的房源（约 664 条）
- `has_coords=true` — 仅返回有经纬度的房源（约 12,117 条，地图模式使用）

（3）数据总览 `GET /api/stats/overview` 的 `by_source` 新增：
```json
{
  "by_source": {
    "anjuke": 12225,
    "lianjia": 526,
    "augmented": 42242
  }
}
```

（4）健康检查 `GET /health` 新增调度器状态：
