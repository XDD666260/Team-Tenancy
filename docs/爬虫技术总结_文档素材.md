## 二、核心问题与解决方案

### 问题1：代理缓存导致翻页数据93.7%重复

**现象**：隧道代理爬取安居客，第2-12页返回内容与第1页完全相同。82,948行数据中仅5,119个唯一标题，每条房源平均重复15.8次。

**根因分析**：

```
翻页请求(page=N) → HTTP GET /cq/sale/{code}/p{N}/
  → 经过隧道代理 → 代理缓存了page=1的响应
  → 后续请求命中缓存 → 返回page=1内容
```

三条根因：
1. 请求**缺少防缓存头**（Cache-Control/Pragma）
2. 翻页URL**模式过于规整**（`/p2/`、`/p3/`），代理以路径前缀作为缓存键
3. **缺乏内容级校验**——解析后未检查页面内容是否与之前页面重复

**解决方案（三层防御）**：

| 层级 | 措施 | 作用 |
|------|------|------|
| 请求层 | `Cache-Control: no-cache, no-store` + URL时间戳`?_t={ts}` | 从HTTP协议层面阻止缓存命中 |
| 重试层 | 换时间戳 → 换代理IP → 直连（3次渐进式重试） | 不同网络路径绕过缓存 |
| 校验层 | 对比本页与上页的房源ID集合，重合度>85%判定为缓存 | 解析后兜底检测 |

**代码关键片段**：

```python
# 1. 防缓存头 (utils.py mobile_get)
headers['Cache-Control'] = 'no-cache, no-store, must-revalidate'
headers['Pragma'] = 'no-cache'

# 2. URL时间戳 (anjuke_fast.py crawl_district)
cache_buster = int(time.time() * 1000) % 1000000
url = f'https://m.anjuke.com/cq/sale/{code}/p{page}/?_t={cache_buster}'

# 3. 内容校验
page_ids = {h['id'] for h in houses}
overlap = len(page_ids & prev_page_ids)
if overlap / len(page_ids) > 0.85:  # >85%重合 → 缓存
    continue  # 换代理重试
```

**效果**：翻页重复率从93.7%降至<5%。

### 问题2：安居客CSS选择器失效

**现象**：2026年6月爬取时，所有房源价格为0。

**根因**：安居客移动站在2026年6月更新了CSS类名。旧选择器`.price`已不存在。

**解决方案**：改用新版CSS选择器：

| 字段 | 旧选择器 | 新选择器(2026.6) |
|------|---------|-----------------|
| 总价 | `.price` | `.content-price`（纯数字） |
| 单价 | `.unit-price` | `.house-avg-price`（元/㎡） |
| 总价+单价 | — | `.price-wrap`（"120万12173元/㎡"） |

**代码**：
```python
# 多级回退策略
price_el = item.select_one('.content-price')        # 新版（优先）
if not price_el:
    pw = item.select_one('.price-wrap')              # 新版包裹元素
    price_text = re.search(r'([\d.]+)万', pw.get_text()).group(1)
if not price_text:
    price_el = item.select_one('.price')             # 旧版兜底
```

### 问题3：深度翻页无新增数据

**现象**：翻到30-50页后每页新增为0，大量浪费代理IP配额。

**根因**：安居客移动站是"软分页"——搜索结果为相关性排序，同一批房源在多个页面重复出现，而非数据库级的严格OFFSET/LIMIT分页。

**数据验证**（渝中区）：

| 页码 | 本页条数 | 新增 | 累计唯一 |
|------|---------|------|---------|
| p1-p3 | 71×3 | ~200 | 200+ |
| p4-p6 | 71×3 | 3 | ~210 |
| p7-p10 | 71×4 | 2 | ~215 |
| p30-p49 | 71×20 | 0 | ~215 |

**解决方案**：实现"连续N页零新增自动停止"（早停优化）：
```python
if new_on_page == 0:
    consecutive_empty += 1
if consecutive_empty >= 3:
    print(f'连续{consecutive_empty}页无新增，停止翻页')
    break
```

### 问题4：私密代理池不稳定

**现象**：部分代理IP请求失败（ProxyError, ChunkedEncodingError），导致区县爬取中断。

**解决方案**：
1. **每次重试换代理IP**：`mobile_get`中每次retry调用`get_proxy()`获取新IP
2. **失败自动刷新代理池**：连续失败时调用快代理API获取20个新IP
3. **API冷却机制**：120秒内不重复调用提取API（节约配额）
4. **页面延迟5-8秒**：降低单IP请求频率

```python
# 代理自动刷新
def mobile_get(url, ...):
    for attempt in range(max_retries):
        chosen_proxy = get_proxy()  # 每次换代理
        ...
        except Exception:
            if attempt >= 2:
                new_count = refresh_proxy_list()  # 刷新代理池

# API冷却
_PROXY_FETCH_COOLDOWN = 120  # 秒
def _should_refetch():
    return time.time() - _PROXY_LAST_FETCH > _PROXY_FETCH_COOLDOWN
```

### 问题5：链家反爬极其严格

**现象**：链家第1页正常返回30条房源，第2页立即触发302重定向到CAPTCHA验证页面（`hip.lianjia.com/captcha`）。

**根因**：链家使用"hip"反爬系统，具备：
1. TLS指纹检测——`requests`库的TLS握手特征被识别
2. Cookie+IP绑定——Cookie与生成时的IP绑定，代理换IP后Cookie失效
3. 行为分析——翻页间隔、Referer链等异常检测

**解决方案**：
1. 使用用户浏览器提取的有效Cookie（含`hip`令牌）
2. 直连模式（不使用代理，避免IP变更导致Cookie失效）
3. 单线程慢爬（页间延迟6-12秒）
4. **接受现实**：仅爬取每区县首页30条（12区县×30=360条）

## 六、技术经验总结

1. **代理选型**：私密代理 > 隧道代理 >> 免费代理。隧道代理有中间缓存层，导致翻页数据重复。

2. **防缓存是爬虫必修课**：Cache-Control头 + URL时间戳 + 内容校验，三道防线缺一不可。

3. **CSS选择器会过期**：目标网站改版时选择器失效，需要多级回退策略和第一页自动诊断。

4. **深度学习成本**：爬虫翻页不是越深越好——安居客10页后基本无新增，早停优化可节约80%代理配额。

5. **链家爬取需要真实浏览器**：requests库的TLS指纹被识别，需使用Playwright/Selenium等方案。
