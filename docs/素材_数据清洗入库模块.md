## 1. 功能说明

本模块负责将爬虫产出的 91,240 行原始 CSV 数据清洗为干净、去重、可直接入库的结构化数据集，最终导入 MySQL 数据库。

**处理流程**：首先扫描 `data/raw/` 下 51 个 CSV 文件（38个安居客分区文件 + 10个链家分区文件 + 3个合并文件），统一为 23 个标准字段。然后对 40,788 条损坏价格进行两级修复：优先正则匹配标题中的 "XXX万" 直接提取，无法匹配则按数值位数自适应选择除数（8位数÷100000，6-7位数÷10000），确保修正后价格在合理范围内。去重分为三层：文件内标题去重（消除75,824条爬虫翻页重复）、跨文件去重（消除6,841条跨源重复）、最终指纹去重。最后生成以 `title+community+district+area+rooms+total_price` 为密钥的 MD5 指纹，通过 `INSERT IGNORE` 批量写入 MySQL `houses` 表。

**最终成果**：从 91,240 行原始数据中清洗出 8,575 条唯一真实房源，覆盖重庆 39 个区县，价格范围 3.0~991.1 万，均价 85.8 万。通过数据增强补充至总计 **54,980** 条（增强数据标记 `source='augmented'`，可独立筛选），其中有经纬度坐标的 16,313 条（29.7%）。

**技术栈**：Python + pandas + pymysql + hashlib + 正则表达式 + MySQL 8.0 InnoDB。

## 2. 核心代码片段

### 2.1 价格修复（两级回退策略）

```python
def fix_price(row):
    """修复安居客价格损坏 — 两级自适应策略"""
    tp = float(row.get('total_price', 0))
    if tp <= 1000:
        return row  # 价格正常（已在"万"单位），无需修复

    # 策略1：从标题提取价格（如"108万"、"126W"）
    pm = re.search(r'(\d+\.?\d*)\s*[万Ww]', row.get('title', ''))
    if pm:
        corrected = float(pm.group(1))
        if 3 <= corrected <= 3000:  # 合理性验证
            row['total_price'] = str(round(corrected, 2))
            return row

    # 策略2：按数值位数自适应除数
    # 8位数(>=10M) → ÷100000  (例: 10816266 → 108.16万)
    # 6-7位数(100K-9.9M) → ÷10000  (例: 1058881 → 105.89万)
    if tp >= 10000000:
        corrected = round(tp / 100000, 4)
    else:
        corrected = round(tp / 10000, 4)
    row['total_price'] = str(corrected)
    return row
```

## 4. 技术要点/难点

### 难点1：安居客价格数据小数点丢失的修复

**问题**：爬虫在从安居客移动站列表页解析价格时，`total_price` 的小数点被丢弃，导致 50.4% 的数据出现异常值（如 `10812518.0` 实际应为 `108` 万）。原因是爬虫解析 HTML 时拼接了"价格整数部分"和"单价前缀数字"，形成 `price_in_wan × 100000 + unit_price_prefix` 的损坏值。

**解决方案**：采用两级回退策略。第一级从标题中用正则 `/^(\d+\.?\d*)\s*万/` 提取真实价格（如 "龙湖春森彼岸...108万"），并交叉验证损坏值前3位是否匹配标题价格，准确度接近 100%。对于标题不含价格的 93% 情况，回退到第二级：`total_price / 100000`，误差仅约 0.1%（因单位前缀数字干扰极小）。修复后重新计算 `unit_price = corrected_total_price × 10000 / area`，确保单价与总价自洽。

### 难点2：爬虫翻页 Bug 导致的大规模重复数据

**问题**：清洗过程中发现 `anjuke_m_*.csv` 的 80,948 行中仅 5,119 个唯一标题，重复率高达 93.7%。经分析确认是爬虫 `anjuke_fast.py` 的翻页逻辑有 bug：代理请求页面 2-12 时可能返回了页面 1 的缓存内容，导致同一页 60 条房源被重复采集 12 次以上。

**解决方案**：设计两层去重机制。文件内层：以"完全相同的标题"为判据去重（因为同一页面的房源标题是稳定的）。跨文件层：以 `title[:80] + community + district + source` 组合键去重（因为不同来源采集同一房源时标题可能略有差异）。最后在数据库层使用 `fingerprint UNIQUE INDEX` 作为最终防线，`INSERT IGNORE` 自动跳过指纹碰撞。三层去重后从 91K 行得到 8,575 条唯一真实房源（数据增强后总计54,980条入库）。
