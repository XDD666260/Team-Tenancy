## 1. 功能说明（200-300字）

本模块实现了重庆市二手房源数据的自动采集，采用双数据源互补策略：以**安居客**（chongqing.anjuke.com）为主力数据源，**链家**（cq.lianjia.com）为辅助数据源。安居客每页展示71条房源信息，反爬门槛较低，适合大规模采集；链家每页30条，反爬机制较为严格，用于补充核心区域的优质数据。

技术实现方面，使用 Python 语言，基于 **Requests** 库发起 HTTP 请求并维护 Session 会话状态，通过 **BeautifulSoup** 解析 HTML 页面，利用 CSS 选择器和正则表达式精准提取房源标题、总价、单价、面积、户型、朝向、装修、楼层、小区名称、商圈等 23 个结构化字段。两个网站的页面结构和选择器完全不同，分别编写了独立的解析函数以适应各自的 DOM 结构。

采集流程为：遍历重庆市主城九区及近郊区县 → 按区县翻页爬取列表页 → 提取每页的房源信息 → 结构化存储为 CSV 文件 → 通过 MySQL 数据库去重与增量更新。翻页时采用随机延迟和 Referer 伪装策略，降低触发反爬机制的概率。截至采集完成，共获取重庆市 XX 区县、超过 XX 条二手房数据（请根据实际运行结果填入数字），覆盖了价格、户型、区域等多维度信息，为后续数据分析和可视化提供了充足的数据基础。

## 2. 核心代码片段（带注释）

### 2.1 安居客列表页解析（主数据源）

```python
def parse_house_info(item):
    """从安居客房源卡片HTML中提取结构化数据"""
    house = {}

    # 标题 — 从 .property-content-title-name 提取
    title_el = item.select_one('.property-content-title-name')
    house['title'] = title_el.get_text(strip=True) if title_el else ''

    # 总价 — 从 .property-price-total-num 提取
    price_text = item.select_one('.property-price-total-num').get_text(strip=True)
    house['total_price'] = float(re.sub(r'[^\d.]', '', price_text))

    # 信息行 — "3室2厅 | 120㎡ | 南 | 精装 | 中层(共32层)"
    # 按 | 分割后逐一解析户型、面积、朝向、装修、楼层
    info_text = item.select_one('.property-content-info').get_text(strip=True)
    parts = [p.strip() for p in info_text.split('|')]

    # 户型解析：正则提取"3室2厅2卫"
    layout = parts[0]
    house['rooms'] = int(re.search(r'(\d+)室', layout).group(1))
    house['halls'] = int(re.search(r'(\d+)厅', layout).group(1))

    # 面积：去除"㎡"和"平米"后转浮点数
    house['area'] = float(parts[1].replace('㎡', '').replace('平米', '').strip())

    house['orientation'] = parts[2]   # 朝向
    house['decoration'] = parts[3]    # 装修
    house['floor_desc'] = parts[4]    # 楼层描述原文
    house['source'] = 'anjuke'        # 数据来源标记
    return house
```

### 2.2 链家列表页解析（辅助数据源）

```python
def get_detail_page(house_id, district_name):
    """访问链家房源详情页，获取完整结构化数据"""
    url = f'https://cq.lianjia.com/ershoufang/{house_id}.html'
    resp = session.get(url, timeout=15)
    soup = BeautifulSoup(resp.text, 'lxml')

    # 总价：.total 元素，去掉"万"字
    total_el = soup.select_one('.total')
    total_price = float(total_el.get_text(strip=True).replace('万', ''))

    # 单价：.unitPriceValue 元素，去除非数字字符
    unit_text = soup.select_one('.unitPriceValue').get_text(strip=True)
    unit_price = float(re.sub(r'[^\d.]', '', unit_text))

    # 小区名：.positionInfo 的第一个a标签
    pos_a = soup.select('.positionInfo a')
    community = pos_a[0].get_text(strip=True)    # 小区名称
    biz_circle = pos_a[1].get_text(strip=True)   # 商圈

    # 经纬度：从页面 <script> 标签的JSON数据中提取
    scripts = soup.find_all('script')
    for script in scripts:
        if script.string and 'resblock' in script.string:
            lng = float(re.search(r'"longitude":\s*([\d.]+)', script.string).group(1))
            lat = float(re.search(r'"latitude":\s*([\d.]+)', script.string).group(1))
            break

    return {
        'title': soup.select_one('h1').get_text(strip=True),
        'total_price': total_price,
        'unit_price': unit_price,
        'community': community,
        'biz_circle': biz_circle,
        'lng': lng, 'lat': lat,
        'source': 'lianjia',
        # ... 更多字段
    }
```

### 2.3 数据去重指纹生成

```python
def make_fingerprint(house):
    """生成房源唯一指纹，用于跨数据源去重"""
    # 基于小区+区县+面积+户型+总价的组合生成MD5哈希
    key = f"{house['community']}|{house['district']}|{house['area']}|{house['rooms']}|{house['total_price']}"
    return hashlib.md5(key.encode()).hexdigest()

# 存储时使用 UNIQUE 索引约束，INSERT 重复数据时自动跳过
# CREATE TABLE houses (..., UNIQUE INDEX idx_fingerprint (fingerprint))
```

## 3. 技术要点/难点

### 难点1：链家反爬机制绕过

**问题描述**：链家采用贝壳集团统一的反爬体系，Python 的 `requests` 库由于 TLS 指纹特征明显，直接请求会被识别并返回人机验证页面（CAPTCHA）。翻页请求尤其敏感，第2页起大概率触发验证。

**解决方案**：
1. ☐ 从浏览器获取真实 Cookie（包括 `lianjia_token`、`lianjia_ssid`、`hip` 等关键字段）
2. ☐ 使用 `requests.Session()` 维持会话状态，请求头完整模拟 Chrome 浏览器
3. ☐ 翻页前先访问区县首页"预热"Session，每次翻页间隔 3~5 秒随机延迟
4. ☐ 设置正确的 `Referer` 头（前一页URL），模拟正常用户浏览行为
5. ☐ 链家每页仅30条，翻页不稳定 → 采用**多区县浅翻页**策略：每区只爬前几页，靠区县数量弥补总量

> 📝 提示：如果你的 Cookie 失效了，重新打开浏览器访问链家 → 通过人机验证 → F12 Console 粘贴 `copy(document.cookie)` → 替换脚本中的 `COOKIE_STRING`

### 难点2：两个数据源的页面结构完全不同

**问题描述**：安居客和链家的 HTML 结构、CSS 类名、数据组织方式完全不同。安居客数据全部在列表页的 HTML 卡片中（`.property-content`），而链家列表页只有房源 ID，需要通过详情页补全房价、经纬度等信息。两个网站的区县 URL 编码也不一致（如璧山区：链家 `bishan` vs 安居客 `bishanqu`）。

**解决方案**：
1. ☐ 分别为两个网站编写独立的解析函数，不共用选择器
2. ☐ 使用浏览器 F12 Console 逐个验证 CSS 选择器（`document.querySelector('.xxx')`）
3. ☐ 链家采用"列表页提取ID → 逐个访问详情页"的两阶段采集
4. ☐ 安居客采用"列表页直接提取全部字段"的单阶段采集（效率更高）
5. ☐ 两源数据合并时，通过指纹（MD5）去重，相同房源以链家数据为准（信息更全）

> 📝 提示：数据源分析笔记中记录了每个网站已验证的 CSS 选择器和区县 URL 编码对照表，修改爬虫前先查阅笔记确认。
