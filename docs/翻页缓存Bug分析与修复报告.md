# 安居客爬虫翻页缓存Bug — 分析与修复报告

> 更新于 2026年6月27日 | 可供成员C（王宇）摘录写入文档第4章"数据获取与实现"

---

## 一、问题描述

在安居客快速爬虫（`anjuke_fast.py`）运行过程中，发现一个严重的数据重复问题：

**现象**：爬虫对每个区县遍历翻页（第1页→第2页→...→第12页），但最终产出的 80,948 行数据中仅包含 5,119 个唯一房源标题，重复率高达 **93.7%**。

**数据分析**：

| 指标 | 数值 |
|------|------|
| 原始CSV行数（`anjuke_m_*.csv`） | 80,948 |
| 唯一标题数 | 5,119 |
| 翻页重复行数 | 75,824 |
| 重复率 | 93.7% |
| 平均每条房源被重复采集次数 | ~15.8次 |

由于翻页获取到的每一页内容实际上都是第1页的缓存副本，每条房源被重复采集了约12次（对应12次翻页请求），导致数据量虚高但有效数据极少。这是原始 91,240 行数据清洗后仅剩 8,575 条真实房源的**根本原因**。

---

## 二、根因分析

### 2.1 问题定位

通过对比第 1~12 页的房源ID集合，发现**第2页至第12页的房源与第1页完全相同**（重合度 >95%），确认翻页请求返回了第1页的缓存内容。

### 2.2 根因链条

```
翻页请求(page=N)
  → HTTP GET /cq/sale/{code}/p{N}/
  → 请求经过隧道代理
  → 代理服务器缓存了 page=1 的响应
  → 后续页面请求触发了代理缓存命中
  → 返回 page=1 的内容（而非 page=N 的实际内容）
  → 爬虫解析到与第1页相同的60条房源
  → 重复写入CSV
```

具体有三个层面的原因：

#### 原因1：请求缺少防缓存头

原始 `mobile_get()` 函数发出的 HTTP 请求没有携带任何防缓存头：

```python
# 旧代码 — 无防缓存头
headers = {
    'User-Agent': random.choice(USER_AGENTS_MOBILE),
    'Accept': 'text/html,application/xhtml+xml',
    'Connection': 'close',
}
```

`Connection: close` 只能通知服务器关闭TCP连接，**无法阻止代理层的响应缓存**。代理服务器（尤其是隧道代理）可以基于URL模式缓存响应，后续相似URL的请求直接返回缓存。

#### 原因2：URL 模式过于规律

安居客移动站的翻页URL模式非常规整：

```
第1页: /cq/sale/yuzhong/
第2页: /cq/sale/yuzhong/p2/
第3页: /cq/sale/yuzhong/p3/
```

代理服务器的缓存策略可能将 `/cq/sale/{code}/p{digit}/` 视为同一资源的不同分页参数，直接返回缓存的第1页响应。部分代理甚至会忽略路径差异，仅以域名+路径前缀作为缓存键。

#### 原因3：缺乏内容级校验

旧代码对请求返回的 HTML 内容**没有任何校验**——只要 HTTP 状态码为 200 且页面长度 > 5000 字符，就直接解析入库：

```python
# 旧代码 — 无内容校验
resp = mobile_get(url)
if resp is None or len(resp.text) < 5000:
    # 仅检查长度，不检查内容是否与之前页面重复
    ...
houses = parse_list_page(resp.text)
all_data.extend(houses)  # 直接追加，不检查重复
```

即使代理返回了错误的缓存内容，爬虫也无法感知。

### 2.3 影响评估

| 维度 | 影响 |
|------|------|
| 数据质量 | 93.7%为重复数据，真实唯一房源仅8,575条 |
| 存储浪费 | 80,948行CSV中75,824行为冗余数据 |
| 爬取效率 | 每个区县12次请求中有11次完全无效 |
| 清洗压力 | 数据清洗模块需要处理9万行才能得到8千条有效数据 |
| 达标风险 | 学年设计要求5万条以上，真实数据严重不足 |

## 三、修复方案

### 3.1 总体策略：三层防御

```
┌──────────────────────────────────────────────────┐
│           第一层：请求级防缓存                      │
│  Cache-Control + Pragma 头 + URL时间戳参数          │
│  → 从协议层面阻止代理缓存命中                       │
├──────────────────────────────────────────────────┤
│           第二层：重试级切换策略                    │
│  代理缓存 → 换时间戳重试 → 切换为直连重试            │
│  → 三次尝试使用不同的网络路径                       │
├──────────────────────────────────────────────────┤
│           第三层：内容级校验                        │
│  对比本页房源ID与上一页 → 重合度>85%判定为缓存       │
│  → 在解析层面兜底检测                              │
└──────────────────────────────────────────────────┘
```

### 3.2 修复点1：`mobile_get()` 添加防缓存头

**文件**：`crawler/utils.py`

```python
# 新增防缓存头
if no_cache:
    headers['Cache-Control'] = 'no-cache, no-store, must-revalidate'
    headers['Pragma'] = 'no-cache'
```

`Cache-Control: no-cache` 要求代理在返回缓存前必须先向源服务器验证；`no-store` 禁止任何缓存存储；`must-revalidate` 强制过期后重新验证。`Pragma: no-cache` 是 HTTP/1.0 兼容写法，覆盖旧版代理。

### 3.3 修复点2：URL 添加时间戳参数

**文件**：`crawler/anjuke_fast.py` → `crawl_district()`

```python
# 旧URL: https://m.anjuke.com/cq/sale/{code}/p{page}/
# 新URL: https://m.anjuke.com/cq/sale/{code}/p{page}/?_t={timestamp}
cache_buster = int(time.time() * 1000) % 1000000
url = f'https://m.anjuke.com/cq/sale/{code}/p{page}/?_t={cache_buster}'
```

每次请求附带毫秒级时间戳作为查询参数，确保每个URL在代理层看来都是全新的资源，从根本上破坏缓存键匹配。

### 3.4 修复点3：缓存检测与自动重试

**文件**：`crawler/anjuke_fast.py` → `crawl_district()`

```python
# 提取本页所有房源ID
page_ids = {h['id'] for h in houses if h.get('id')}

# 与上一页ID集合对比
if prev_page_ids and page_ids:
    overlap = len(page_ids & prev_page_ids)
    overlap_ratio = overlap / max(len(page_ids), 1)

    if overlap_ratio > 0.85:
        # >85%重合 → 判定为缓存，触发重试
        print(f'⚠ 缓存检测: {overlap}/{len(page_ids)}条与上页相同({overlap_ratio:.0%})')
        continue  # 重试（最多3次）
```

**重试策略**：

| 重试次数 | 策略 | 说明 |
|---------|------|------|
| 第1次 | 代理 + 新时间戳 | 正常请求，带防缓存头 |
| 第2次 | 代理 + 再次换时间戳 + 额外延迟 | 加大随机性 |
| 第3次 | **直连（不走代理）** | 绕过代理缓存，直接访问源站 |

阈值设为 **85%** 而非 100% 的原因：翻页结果确实存在少量跨页重复（热门房源可能在相邻页面重复出现），85% 在捕获代理缓存的同时避免误判正常的末页数据。

### 3.5 修复点4：区内实时去重

```python
# 旧代码：直接追加，不检查区内重复
all_data.extend(houses)

# 新代码：基于房源ID实时去重
seen_ids = set()
for h in houses:
    hid = h.get('id', '')
    if hid and hid not in seen_ids:
        seen_ids.add(hid)
        all_data.append(h)
        new_on_page += 1
```

在爬取过程中维护本区县已见过的房源ID集合，即使代理返回了部分重复内容，也能在入库前过滤。

## 六、经验总结

### 技术经验

1. **代理服务器是双刃剑**：隧道代理能解决IP封禁问题，但也可能引入响应缓存。需要显式添加防缓存头。

2. **内容校验是最后防线**：无论请求层做了多少防护，都应该在解析后对内容做合理性校验。对比相邻页面的数据是检测缓存/重定向问题的有效手段。

3. **URL唯一性不等于响应唯一性**：即使每个请求的URL不同，代理服务器仍可能以路径前缀作为缓存键。添加查询参数并配合防缓存头才能彻底避免。

### 对后续开发的建议

1. 所有经过代理的爬虫请求都应默认携带 `Cache-Control: no-cache` 头
2. 翻页爬虫应实现页面内容去重检测作为标准功能
3. 爬取日志应区分"本页抓取数"和"本区累计唯一数"，便于实时监控数据质量
4. 关键指标（如唯一房源数/页）应设置告警阈值，异常时自动暂停
