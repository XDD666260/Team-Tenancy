## 一、功能说明

本模块实现了重庆二手房数据的**定期增量更新机制**，确保数据库中的房源信息保持时效性。

**核心功能：**
- **指纹去重机制**：链家数据使用 `lj_{source_id}` 作为指纹，安居客数据使用 `MD5(title+community+district+area+rooms+price)` 作为指纹。每条新抓取的房源与数据库现有指纹比对，判断"新增/价格变动/不变"三种状态。
- **增量爬虫** (`crawler/update.py`)：支持按数据源(lianjia/anjuke)、按区县分别增量抓取，只爬列表页前几页（最新上架房源），对比后仅插入新增和更新变动，无需全量重爬。
- **下架标记**：超过30天未更新的 `on_sale` 房源自动标记为 `probably_sold`。
- **API触发**：FastAPI 路由 `POST /api/update` 一键触发后台增量更新，`GET /api/update/status` 实时查询更新进度。
- **爬取日志**：每次更新写入 `crawl_log` 表，方便追踪数据变化和排查问题。

**技术栈：** Python + pymysql + requests + BeautifulSoup + FastAPI。

def update_single_listing(cursor, source, row):
    """处理单条房源：与DB对比，决定新增/更新/跳过"""
    fingerprint = make_fingerprint(source, row)
    if not fingerprint or fingerprint in ('lj_', 'ajk_'):
        return 'skipped'

    # 查询是否已存在（指纹唯一索引保证O(1)查找）
    cursor.execute(
        "SELECT id, total_price, unit_price, status FROM houses WHERE fingerprint=%s",
        (fingerprint,)
    )
    existing = cursor.fetchone()

    if not existing:
        # 🆕 新房源 → INSERT
        cursor.execute(f"""
            INSERT INTO houses (title, district, ..., fingerprint)
            VALUES (%s, %s, ..., %s)
        """, values + (fingerprint,))
        return 'new'
    else:
        db_id, old_price, old_unit, old_status = existing
        new_price = float(row.get('total_price', 0) or 0)
        # 价格变动 > 0.1万 → UPDATE
        if abs(new_price - float(old_price or 0)) > 0.1:
            cursor.execute("""
                UPDATE houses SET total_price=%s, unit_price=%s, last_updated=NOW()
                WHERE id=%s
            """, (new_price, new_unit, db_id))
            return 'updated'
        return 'skipped'

## 四、技术要点/难点

### 难点1：反爬虫机制与代理IP

**问题**：安居客/链家均有较强的反爬机制。实测中，第一次请求可成功抓取71条房源（CSS选择器 `li.item-wrap` 验证正确），但第二次请求即被拦截，返回 58同城 `antibot` 验证页面（724字节空白页）。链家则需要有效的登录Cookie，Cookie过期后返回空页面（1119字节）。

**解决方案**：
- 安居客使用**隧道代理**（如快代理），每次请求自动更换IP，绕过频率限制
- 链家使用**浏览器Cookie** + UA轮换 + 页间长延迟（30-150秒），模拟人类浏览行为
- 反爬检测代码：`if 'antibot' in resp.text or 'xxzlGatewayUrl' in resp.text: 被拦截`
- 增量爬取每区县只抓3页（约200条），比全量爬取对目标网站压力更小

**已验证的完整流程**：
```bash
# 方式1（推荐）：使用已有代理爬虫 → 增量入库
python crawler/anjuke_fast.py          # 代理爬虫，产出CSV
python backend/update_listings.py       # CSV增量导入数据库

# 方式2：直接增量（需配置代理）
# 在 update.py 中设置 PROXY_URL 后运行
python crawler/update.py --source anjuke --max-pages 3
```

### 难点2：指纹去重策略的平衡

同上（参见上一版文档）。

**问题**：不同的数据源（链家 vs 安居客）对同一套房源的标题、价格描述格式完全不同，无法用同一种指纹算法。如果指纹太宽松（仅基于标题），会导致不同房源误判为同一套；如果指纹太严格（基于所有字段），同一房源稍有变动就无法匹配。

**解决方案**：
- 链家数据有唯一 `source_id`（数字ID），直接使用 `lj_{source_id}` 作为指纹，精确度100%。
- 安居客无稳定ID，使用 `MD5(title前60字符 + community + district + area + rooms + total_price)` 生成指纹，既保证唯一性又能容忍标题微调。
- 指纹字段设为 `UNIQUE INDEX`，数据库层面保证不重复。

### 难点2：全量爬取 vs 增量爬取的取舍

**问题**：如果每次全量爬取所有区县所有页面，耗时数小时且会给目标网站造成压力。如果只爬第一页，可能遗漏新上架房源。

**解决方案**：
- 增量模式只爬每个区县前3页（约90条最新房源），与数据库对比后仅处理新增和变动。
- 每页请求间隔1秒（`time.sleep(1)`），避免被反爬。
- 通过 `last_updated` 字段追踪数据新鲜度，超过30天未更新的自动标记下架。
- 全量回填使用 `anjuke_backfill.py` 单独处理，与日常增量更新分离。
